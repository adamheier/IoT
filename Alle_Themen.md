## Ubiquitous (Pervasive) Computing

## Allgegenwärtigkeit (Durchdringung)

Sehr hohe Anzahl digitaler Endgerate,integriert in Altagsleben ohne dass sich Nutzer dessen zwingend bewusst sind

Zumeist (unsichtbar) eingebettet in stationäre oder mobile Altagsgegenstande z.B. Haushaltsgerate, Kleidung, Buroausstattung etc.

- Häufig ansprechbar über Smartphones oder Voice Control

## Vernetztheit

Endgerate sind mit anderen Endgeräten und Anwendungen verbunden (uber das Internet)

Kommunikation häufig durch funkbasierte aber auch durch kabelgebundene Ubertragungstechnologien

## Digitaler Zwilling (Digital Twin)

besitzt bidirektionale,automatische Datenverbindung zwischen virtuellem und physischem Objekt

- zeigt (sofort) Veränderungen im Zustand des physischen Objekts an

- fuhrt Datenverarbeitung durch

## Digitaler Faden (Digital Thread)

verbindet verschiedene Datenquellen entlang des Lebenszyklus eines physischen Objekts

ermoglicht Datenverbindung zwischen physischem und virtuellem Objekt, jedoch ist bidirektionaler Datenfluss nicht zwingend erforderlich

fuhrt keine Datenverarbeitung durch

## Typen Digitaler Zwillinge (2)

## Digitaler Schatten (Digital Shadow)

- hochgenaue Darstellung eines physischen Objekts (Industrieprodukts)

Ziel, ein Echtzeitbild auf Basis relevanter Daten zu erstellen

- jedoch keine automatischen bidirektionalen Datenverbindungen

- Datenverarbeitung ist nicht zwingend

## Digitales Model (Digital Model)

kann manuelle Datenverknüpfung zwischen virtuellem und physischem Objekt enthalten, muss aber nicht

repliziert keine Zustandsänderungen des physischen Objekts

- fuhrt keine Datenverarbeitung durch

## Data Streaming und Data Stream Management Systeme

## □ Data Stream

kontinuierlicher Fluss von Daten ohne zeitlich begrenztes Ende

z.B. Sensordaten, Benutzerinteraktionen auf Websites, Finanztransaktionen, Social-Media-Aktivitaten, Log-Dateien von IT-Systemen

zeitlich geordnete Reihenfolge

. z.B.dokumentiert mit Zeitstempeln im Datenstrom

- variable Datenrate mit meist nur sequentiellen Zugriffsmöglichkeiten

## Data Stream Management Systeme

auch Stream Processing oder Event Streaming Systems genannt

- dienen zur effizienten Verwaltung, Verarbeitung und Analyse von Datenstromen

erzeugen i.d.R. neue Datenstrome

. Input Stream → Verarbeitungslogik → Output Stream

gekennzeichnet durch skalierbare Architekturen und leistungsfahige Algorithmen

mit Echtzeitverarbeitung, niedriger Latenz und Fehlertoleranz

<!-- image-->

## Zentrale Funktionen von Data Stream Management Systemen (1)

## Continuous Queries

- überwachen den eingehenden Datenstrom und wenden spezifizierte Filter-, Transformations- und Aggregationsoperationen an

- aktualisieren die Ergebnisse in Echtzeit aktualisiert, sobald neue Daten eintreffen

bleiben aktiv,bis sie explizit inaktiviert oder geloscht werden

## Event-Time Processing

Verarbeitung von Ereignissen basierend auf dem Zeitpunkt, zu dem sie tatsächlich aufgetreten sind (→ Ereigniszeit)

nicht auf Zeitpunkt, zu dem sie vom System empfangen werden (→ Systemzeit oder Eingangszeit)

essentiell in verteilten Systemen, in denen Ereignisse mit variabler Latenz eintreffen konnen (→ u.a. loT)

## Zentrale Funktionen von Data Stream Management Systemen (2)

## Windowing

Unterteilung des Datenstroms in endliche Abschnite (Zeitfenster), um darauf Analysen und Aggregationen durchzufuhren

Sliding Windows (Gleitende Fenster): sich überlappende Fenster

Ereignisse konnen in mehreren Fenstern vorhanden sein

ermoglicht feinere Granularitat bei der Analyse von Datenstromen oder Zeitintervalen

. jedes Ereignis im Datenstrom gehort genau Zu einem Fenster

eignet sich fur periodische Analysen über festgelegte Zeitintervalle

## Representational State Transfer (REST)

## ■ Architekturansatz fur Kommunikation zwischen Client und Server in verteilten Systemen

Programmierparadigma zur (einfachen) Realisierung von Web Services

Alternative zu SOAP,WSDL und RPC

. Realisierung sog. RESTful Anwendungen

Weder Protokoll noch Standard, sondern:

Nutzung Von Standards wie HTTP/S, URI, HTML, JSON oder XML

· Jede REST-Nachricht enthalt alle Informationen, die fur Server bzw. Client notwendig sind, um Nachricht zu interpretieren (Zustandslosigkeit)

## Verwendung moglichst vieler HTTP-Methoden, z.B.

GET: Lesen (Anfordern) einer Ressource vom Server

- POST: Anlegen (Einfugen) einer Ressource auf dem Server

- PUT: Andern einer Ressource auf dem Server

DELETE: Loschen einer Ressource auf dem Server

OPTIONS: Prufung, welche Methoden fur eine Ressource vom Server angeboten werden

## HTTP-basiertes Protokoll fur Client-Server-Kommunikation

ursprunglich von Microsoft veroffentlicht, mittlerweile von vielen Softwareherstellern unterstutzt (z.B. SAP mit Fiori und SAP Gateway)

:basiert auf REST-Architekturansatz
